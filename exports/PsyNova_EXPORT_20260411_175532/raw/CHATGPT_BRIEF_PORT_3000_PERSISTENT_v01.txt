CONTEXT
--------------------------------------------------------------------------------
- Stack: NestJS backend listens on 0.0.0.0:3000 (PORT env, default 3000).
- Docker Compose maps the same host port: backend service uses "3000:3000"
  (container name often: psynova-backend).
- Developers also run locally: `cd backend && npm run start:dev` (nest --watch).

OBSERVED ISSUE
--------------------------------------------------------------------------------
- Error: `listen EADDRINUSE: address already in use 0.0.0.0:3000`
- Nest logs show routes mapped, then immediate failure to bind — a second process
  (or Docker-published service) already owns :3000.
- Manual `fuser -k 3000/tcp` or `docker compose stop backend` fixes it until the
  next time both paths run again. Problem returns after reboot if Docker auto-starts
  stacks or user scripts start duplicate listeners.

ALREADY IN REPO (mitigation, not permanent policy)
--------------------------------------------------------------------------------
- Script: `ops/free_port_3000.sh` — detects Docker publishing :3000 and suggests
  `docker compose stop backend`; otherwise tries fuser/lsof on host.
- Not a substitute for an architectural rule that survives reboots.
